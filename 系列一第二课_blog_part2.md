# 系列一 写 OS 内核你需要想清楚的四件事
## 第二课 | 多核并发的深水区：内存模型、MESI 与原子操作的底层逻辑

> 接上篇：我们讨论了单核系统的并发机制。但多核处理器才是真正复杂的地方——多个 CPU 核心真正同时运行，传统的“关中断”保护机制完全失效。

在多核系统中，内核开发者必须面对三个层面的核心挑战：数据竞争、缓存一致性以及内存可见性与重排序。

---

## 1. 多核处理器的缓存架构

典型的多核处理器采用分层缓存结构：每个核心拥有独立的 L1 缓存（分为数据缓存和指令缓存），同一簇内的核心共享 L2 缓存，所有核心共享末级 L3 缓存，最终统一访问主存。

## 2. 乱序执行与内存模型

编译器和 CPU 为了性能会对指令进行重排序。在单核环境下，这通常不会引发问题；但在多核环境下，**一个核心试图观察另一个核心的内存操作顺序，而这个顺序可能已经被本地重排**。

![多核环境下乱序执行引发的内存可见性问题](image-1.png)
*图 2：多核环境下乱序执行引发的内存可见性问题示意图*

### 不同架构的内存序保证
- **强内存模型（如 x86）**：硬件默认提供较强的顺序保证，唯一需要显式屏障的场景是防止 `Store -> Load` 重排（常用 `mfence`）。
- **弱内存模型（如 ARM/RISC-V）**：默认允许更激进的重排序，开发者必须手动插入屏障指令（如 ARM 的 `dmb`, RISC-V 的 `fence`）来确保正确性。

## 3. 缓存一致性与 MESI 协议

现代 CPU 每个核心都有独立的缓存。核心 A 修改了某个变量，核心 B 的缓存里可能还是旧值。

![典型多核处理器缓存拓扑结构](image-3.png)
*图 3：物理上的缓存隔离导致逻辑上的不一致*

**MESI 协议**是解决此问题的基石，它通过四个状态（Modified, Exclusive, Shared, Invalid）管理缓存行的一致性。当 Core A 写入一个处于 `Shared` 状态的缓存行时，会广播 `Invalidate` 消息，使其他核心的该缓存行变为 `Invalid`。

## 4. 原子操作的底层实现

原子操作是大多数锁的基石。不同架构实现原子操作的方式迥异：

### ✅ x86 / x86_64
通过 `lock` 前缀确保多核环境下操作的原子性，提供丰富的内置原子指令如 `cmpxchgq`。

![x86 上的 64 位 CAS 实现示例](image-6.png)
*图 5：x86 上的 64 位 CAS 实现示例*

### ✅ ARM / RISC-V
采用 **LL/SC (Load-Linked/Store-Conditional)** 范式。

![ARM 架构下的 LL/SC 实现](image-7.png)

![RISC-V 架构下的 LL/SC 实现](image-8.png)

## 5. 内核中的并发处理策略

- **percpu 机制**：划定每个 CPU 专属的内存区域，从根本上消除竞争。
- **各类锁**：spinlock、mutex、rwlock 等。
- **无锁机制**：基于原子操作的无锁数据结构，适合高竞争场景。

> 💡 **建议**：多核下暴露的 bug 往往最难调试（Heisenbug）。**从设计阶段就把并发安全纳入考虑，而不是事后补锁。**
