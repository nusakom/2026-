# 系列三 | 拔靴自举：内核如何在“一无所有”的环境中启动自己？

> 接上篇：我们讨论了多核并发的挑战。本篇我们将目光转向内核启动的关键阶段——自组织性。

所有应用依赖库和内核，库也依赖内核，但内核依赖谁？**内核依赖它自己。**
这就是自组织性问题的本质：内核必须能够在没有任何外部支撑的情况下，把自己"拔靴带"式地启动起来。

---

## 1. 鸡生蛋还是蛋生鸡

内核启动时面临着许多类似的悖论。其中最典型的是内存管理。

### 内存分配器的自举困境
内核启动时需要一个内存分配器来管理内存，但构造内存分配器本身就需要分配内存。

**常见解法：两阶段自举**
1. **第一阶段**：在编译期预留一块静态内存区域，并在这块内存上构造一个简单的"早期分配器"（early allocator）。
2. **第二阶段**：早期分配器就绪后，再将内容迁移到正式的内存管理框架（如 buddy system + slab）。

Linux 的做法是通过 `memblock` 解决早期自举问题，再逐步过渡到完整分配器。

## 2. 页表的自举困境

页表本身也需要占用物理内存。当内核需要建立虚拟地址映射时，可能需要新建页表项，而新建页表项需要分配物理页面，分配物理页面又依赖内存分配器，内存分配器又需要映射已就绪……

```
建立映射 → 需要分配页表页 → 需要内存分配器 → 需要映射已就绪 → ...
```

**常见解法**：在启动汇编代码中，手动在固定内存区域构造一个初始页表，使用 **direct mapping**（虚拟地址 = 物理地址 + 固定偏移），让内核先"站稳脚跟"，再逐步迁移到正式框架。

> 💡 **注意**：在资源受限的开发板上，可以把所有物理内存一次性线性映射进内核。但现代服务器物理内存动辄数 TB，启动时无法预先映射全部地址空间，必须按需动态分配页表。这是工程妥协与通用方案的权衡。
